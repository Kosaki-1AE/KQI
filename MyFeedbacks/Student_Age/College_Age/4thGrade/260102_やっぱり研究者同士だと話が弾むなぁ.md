ここまで来て気づいたわけですわ。構造って動かせるよね？って。
- 判断粒度（どこまで精度を出すか）
- トランジションの閾値

んで何となく思いついたんだけどさ。エネルギーを構造にするっていうことをさw
簡単にいうと構造化ってのができることがめっちゃ重要なのよね、俺にとってのダンスがこれに該当するんだけどさ。
- エネルギー＝流すもの → **配置するもの**：**地形（谷・流路）を作る**感じ
- 制御が「操作」から「性質」になる：Bufferそのものが制御則になる
- 効率の概念が変わる：効率＝**どれだけ無視しても回るか**
「主制御は点、下位は場」ってなるっぽい。

そんでさ、Bufferが机と同じようなもんだから、何もない環境とかだけでも「その場の処理効率」については向上するわけよね。Bufferって「その場で処理せなあかん因数分解の分解能」なんでね。
この勢いで全部言い換えようか。これだと分かりやすいだろうし。
俺目線：GBMCでのB(Buffer層)から見てる映像です⇩
- CPU：即時判断核 / 一時引き受け場所
- RAM：LIVE Buffer / ガチその場で処理せなあかん因数分解能
- SSD：遅延責任倉庫 / アクセスに情報量が1bit以上必要＝コストが確定で発生する
- Process：未回収の判断列 / 既に引き受けると決めた責任なので、終わらせないとコストが面倒くさい
- OS：責任配分系 / ここで進められるか否かを自分で決める必要があるんだよねぇ
- Network：外部責任流路 / 自分の外に判断を投げる経路で、レイテンシ＝信頼コストになります
- Chache：反射済み判断 / 即興をするための型(もっというとこれが**武道での型**になります)
- I/O Device：責任注入点 / まぁ因果発生装置ってことなので、あんま触りすぎひんようにね。
- Compiler：責任変換器 / 曖昧な情報から実行可能な責任単位へ変換することそのものですわ
	- 字句解析：意味の粒度切り / "意味の因数分解"の開始点
	- 構文解析：構造チェック / これ通らんと型比較ができないんすわ、実際には入力とかChacheとか色々なものを照合するってのを一気にやってます
	- 意味解析：責任の整合性検査 / 対象の責任が背負えるか
	- 最適化：Buffer調整 / 無駄なコストを削って余計なノイズを減らす
	- コード生成：実行可能な一手に変換 / 1 or 0が確定するラインすねぇ
⇨ 静かな環境＝構造解析精度が爆上がりする＝位相アンテナの感度が上がる＝色々見たり聞いたりを繰り返し始める

Buffer層で見てるから、いくらでも机の配置場所を変える＝構造を圧縮するってことができる＝覚え方が分かれば無双できるってことかw
そんでアンテナ合わせ＝位相合わせを行うのが最後の課題かぁ...。ここでのアンテナ合わせってのが早いと**才能ある**認定受けるんだけどね。
